106 /*
107  * Here we define mpack_assert() and mpack_break(). They both work like a normal
108  * assertion function in debug mode, causing a trap or abort. However, on some platforms
109  * you can safely resume execution from mpack_break(), whereas mpack_assert() is
110  * always fatal.
111  *
112  * In release mode, mpack_assert() is converted to an assurance to the compiler
113  * that the expression cannot be false (via e.g. __assume() or __builtin_unreachable())
114  * to improve optimization where supported. There is thus no point in "safely" handling
115  * the case of this being false. Writing mpack_assert(0) rarely makes sense;
116  * the compiler will throw away any code after it. If at any time an mpack_assert()
117  * is not true, the behaviour is undefined. This also means the expression is
118  * evaluated even in release.
119  *
120  * mpack_break() on the other hand is compiled to nothing in release. It is
121  * used in situations where we want to highlight a programming error as early as
122  * possible (in the debugger), but we still handle the situation safely if it
123  * happens in release to avoid producing incorrect results (such as in
124  * MPACK_WRITE_TRACKING.) It does not take an expression to test because it
125  * belongs in a safe-handling block after its failing condition has been tested.
126  *
127  * If stdio is available, we can add a format string describing the error, and
128  * on some compilers we can declare it noreturn to get correct results from static
129  * analysis tools. Note that the format string and arguments are not evaluated unless
130  * the assertion is hit.
131  *
132  * Note that any arguments to mpack_assert() beyond the first are only evaluated
133  * if the expression is false (and are never evaluated in release.)
134  *
135  * mpack_assert_fail() and mpack_break_hit() are defined separately
136  * because assert is noreturn and break isn't. This distinction is very
137  * important for static analysis tools to give correct results.
138  */
